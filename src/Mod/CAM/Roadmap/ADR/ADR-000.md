
# ADR-000: Ubiquitous Language

## Context

Software doesn't cope well with ambiguity. In order to have productive conversation with other human developers or with AI coding agents, consistent use of common and rigorous language is needed. The concept of a 'Ubiquitous Language' is described in Eric Evan's book Domain Driven Design

## Decision

This ADR will contain the Ubiquitous Language Glossary.  It will be updated as needed.

## Language

### Job & containers

**Job**:
The top-level CAM document object that owns a single machining project. Holds the **Model**, **Stock**, **Operations**, **Tool Controllers**, **Setup Sheet**, **Post-Processor** selection, and **Machine**.

**Job Template**:
A saved blueprint of a **Job**'s configuration (post-processor, fixtures, output settings, setup-sheet defaults, tool controllers, stock, machine reference) for reuse on new projects. Serialized via the `JobTemplate` class as XML attribute strings. Distinct from a **Machine Template** — different file format and scope.
_Avoid_: bare "template" (always qualify as **Job Template** or **Machine Template**).

**Model**:
The part being machined. Referenced by the Job through a **Resource Clone** so the original geometry is not mutated.
_Avoid_: Part, workpiece (in code/UI — "Model" is the canonical name).

**Stock**:
The raw material the **Tool Bit** cuts away from. Visualization-only — does not constrain toolpaths. Stock has a type: `FromBase` (derived from the **Model** with offsets — "base" here = the **Model**, not **Base Geometry**), `CreateBox`, `CreateCylinder`, or `NoStock`.
_Avoid_: blank, billet, raw.

**Resource Clone**:
A `Draft.clone` of a base object tagged with a `PathResource` property identifying its role (e.g. `"Model"`). Used so the **Job** has its own reference to geometry without owning it.

**Setup Sheet**:
A container on the **Job** holding default values and **Job**-specific defaults (rapid speeds, safe-height expressions, depth expressions, fixtures). New **Operations** read defaults from here.
_Avoid_: defaults file, config (these are domain-vague).
_Note_: clashes with the industry term "setup sheet" (a printed shop-floor document describing how to set up a part). In this codebase, "Setup Sheet" always means the defaults container — never the shop-floor document. The closest analogue to the industry meaning is the **Sanity Check** report.

**Fixture**:
A work-coordinate-system selection (G54–G59.9) under which a set of **Operations** is output. A **Job** can output multiple fixtures.
_Avoid_: WCS (used in code occasionally), work offset, clamping (this is *not* the physical workholding).
_Note_: clashes with the industry term "fixture" meaning the physical workholding device. In this codebase, "Fixture" always means the WCS. A future initiative may revisit this naming.

### Operations

**Operation** (also **Op**):
A single machining step that produces a **Toolpath** — e.g. a pocket, a profile, a drilling cycle. **Op** is a synonym, used as code shorthand (filenames, variable names, `Path/Op/`). Inherits from the internal base class `ObjectOp` (`Path/Op/Base.py`). Configured by **Feature** flags + properties + **Base Geometry** + a **Tool Controller**.
_Avoid_: step, task, action. `ObjectOp` is internal — not a domain term.

**Feature** (in operation context):
A bit-flag (`FeatureTool`, `FeatureDepths`, `FeatureHeights`, …) that an **Operation** ORs together to declare which standard properties it supports. Distinct from FreeCAD's generic notion of "feature" (a document object).

**Base Geometry**:
The user-selected geometry on the **Model** (faces, edges, vertices) that an **Operation** acts on. Stored on the operation as the `Base` property. Canonical term — the bare word "Base" is *not* a domain term and only appears as code-internal naming (`Path/Base/`, `ObjectOp.Base`, `FromBase`).
_Avoid_: target, selection, input geometry, "Base" alone.

**Drillable**:
A predicate applied to a candidate face. A face is *drillable* with respect to a given **Tool Bit** diameter and an approach vector (machine pose). The `Path/Base/Drillable.py` module supplies the classifier predicates (e.g. `isDrillableCylinder`) used by **Circular Hole** operations to filter selectable **Base Geometry**.
_Avoid_: "drill candidate", "hole feature" (those are not the predicate).

**Depth profile** (cluster of properties on most ops):
- **Start Depth** — Z height where cutting begins.
- **Final Depth** — Z height where cutting ends.
- **Step Down** — Increment **axial to the tool** (along the tool's own axis) between successive depth passes. For planar ops this is a Z decrement per pass; for **Rotary Surface** this is a *radial* decrement (each layer moves the cutter closer to the rotary axis). The geometric direction differs but the semantic is the same: each pass advances the cutter one step further through the material along the tool axis.
- **Step Over** — Increment **radial to the tool** (perpendicular to the tool's axis, across the surface) between adjacent passes. For planar facing/pocketing this is the lateral spacing between parallel passes; for **Rotary Surface** this is the *axial* advance per full revolution (the spiral pitch). The geometric direction differs but the semantic is the same: each pass shifts the cutter sideways by one tool-radial increment.
- **Finish Depth** — the amount of material reserved for the **Finish Pass**. Controlled by the `FinDepth` expression or property on the operation. Zero means no separate finish pass.
- **Clearance Height** — Z for safe rapids over fixtures.
- **Safe Height** — Z for rapids within the operation.

_Avoid_: "axial stepover", "radial stepdown" (qualifier is redundant — Step Over is always along-surface, Step Down is always into-material). For rotary ops, just say **Step Over** and **Step Down**; the axial/radial mapping is implicit.

**Finish Pass**:
A final cutting pass that follows a new toolpath offset from the roughing passes, removing only a thin, controlled amount of material (the **Finish Depth** when depth-based, or a lateral allowance when offset-based). The purpose is surface quality: a light, consistent cut produces a better finish than the variable remainder left by roughing. A Finish Pass is *not* a repetition of the previous pass — it has its own geometry.
_Avoid_: conflating with **Spring Pass**.

**Spring Pass**:
An *exact repetition* of the immediately preceding cutting pass with no change in depth, offset, or feed. Its purpose is deflection recovery: the first pass leaves a thin witness of uncut material because tool and workpiece deflect under load; the repeat pass, seeing almost no load, springs back to true and cleans the witness. Not currently implemented as a first-class feature in the CAM workbench, but the concept is distinct from **Finish Pass**.
_Avoid_: conflating with **Finish Pass**. A **Finish Pass** follows a *new* toolpath with a controlled allowance; a **Spring Pass** re-runs the *same* toolpath to recover deflection.

**Operation catalog** (current operation types):
- Pocket — 2.5D area clearing inside a boundary. Canonical implementation is `Path/Op/PocketShape.py`. `Path/Op/PocketBase.py` is an internal shared base class (also used by Mill Face, Engrave) and is not a user-facing concept.
- 3D Pocket — 3D cavity clearing. Implemented in `Path/Op/Pocket.py`. Distinct from **Pocket**.
- Profile — cuts along a contour (inside/outside/on-line).
- Adaptive — high-efficiency clearing using a constant tool engagement strategy.
- Drilling — canned-cycle drilling on circular holes.
- Tapping — thread cutting in holes.
- ThreadMilling — milled thread on a hole.
- Helix — helical interpolation into a hole.
- Engrave — single-line engraving along edges.
- Vcarve — V-bit carving using a Voronoi medial-axis.
- Slot — slot cuts.
- Surface — 3D surface finishing (requires **OCL**).
- Waterline — constant-Z surface finishing (requires **OCL**).
- Mill Facing — face-milling the top of stock. Canonical implementation is `Path/Op/MillFacing.py`; `Path/Op/MillFace.py` is legacy and slated for deprecation.
- Probe — probing routine.
- Deburr — edge-break pass.
- Custom — user-supplied G-code injected as a path.

### Tools

**Tool** (bare):
Not a domain term. Always disambiguate to **Tool Bit**, **Tool Controller**, **Tool Library**, or **Tool Shape**. The compound names below are the only canonical forms.

**Tool Controller** (also **TC**):
The runtime pairing of a **Tool Bit** with spindle speed, feed rates, and rapids. **Operations** reference a **Tool Controller**, not a **Tool Bit** directly. Lives under the **Job**'s `Tools` group.
_Avoid_: tool slot, tool assembly (in CAM-workbench code; "Tool Controller" or "TC" only).

**Tool Bit**:
The cutter itself — geometry, cutting edges, dimensional parameters. Has a **Tool Shape** classifier: `Endmill`, `Ballend`, `Bullnose`, `Chamfer`, `Drill`, `VBit`, `Dovetail`, `Reamer`, `SlittingSaw`, `Tap`, `ThreadMill`, `Probe`, `TaperedBallNose`, `Radius`, `Custom`.
_Avoid_: cutter, bit, end mill (when speaking of the abstraction — use the specific shape name when speaking of the kind).

**Tool Library**:
A persisted collection of **Tool Bits**, serialized to disk. Independent of any **Job** — libraries are shared across projects.

**Tool Shape**:
The geometric template for a **Tool Bit**. Defines the parametric shape used to render and (for some operations) compute engagement.

### Toolpath generation

**Toolpath**:
A sequence of motion commands (`G0`/`G1`/`G2`/`G3`/…) attached to an **Operation**. The output of `opExecute()`. Canonical user-facing and docs term.
_Avoid_: trajectory, motion plan, "path" (lowercase, as a noun).

**Path** (namespace only):
The Python namespace `Path.*` and source tree `Path/`. Legacy name, retained for backwards compatibility after the Path → CAM workbench rename. Not a domain term — never use bare "Path" to mean **Toolpath** in docs or UI. The `PathScripts.*` namespace (e.g. `PathScripts/PathUtils.py`) is also legacy and is being migrated into `Path/`; treat it as code-internal only.

**Generator**:
A low-level motion-pattern function in `Path/Base/Generator/` (drill, helix, tapping, spiral facing, zigzag facing, linking, etc.). **Operations** call generators to produce specific motion segments.
_Avoid_: strategy, planner, G-code generator (use **Post-Processor** for that — never conflate the two).

**Dressup**:
A wrapper object that takes an **Operation**'s **Path** and modifies it in-place: holding **Tags**, **Boundary** trimming, **DogboneII** corner relief, **Array** duplication. A dressup *replaces* its child operation in the **Job**'s output sequence.
_Avoid_: post-modifier, path filter, decorator.

**Tag** (holding tag):
A small bridge of uncut material left on a **Profile** so the part doesn't move when fully cut through. Created via the **Tags** **Dressup**.
_Avoid_: tab (industry alternate; we use "tag").

### Output pipeline

**Post-Processor** (also **Post**):
The component that converts a **Job**'s **Paths** into machine-specific G-code text. Selected per-**Job**. Lives in `Path/Post/scripts/` as `<machine>_post.py`.
_Avoid_: G-code generator, exporter.

**Machine**:
A description of the physical CNC machine: axes (linear/rotary, table/head), units, limits, plus post-processor formatting settings (line numbers, comment style, header content). Configured via the **Machine Editor** UI and saved as `.fcm` files.
_Avoid_: controller (overlaps with Tool Controller), CNC.

**Machine Template** (`.fcm` file):
JSON serialization of a **Machine** definition. Templates ship in `Machine/machines/` and are loaded into a **Job**. Distinct from a **Job Template** — different file format and scope.
_Avoid_: machine config, machine profile, bare "template".

**Sanity Check** (also **CAMSanity**):
A pre-output validation pass over a **Job** that produces an HTML **Sanity Check report** flagging missing tools, unset feeds, suspicious depths, etc. Canonical user-facing term.
_Avoid_: linter, validator, pre-flight check, job report.

## Relationships

- A **Job** has exactly one **Model**, one **Stock**, one **Setup Sheet**, one **Machine**, one **Post-Processor**, and many **Operations** and **Tool Controllers**.
- An **Operation** references exactly one **Tool Controller** (when it has `FeatureTool`).
- A **Tool Controller** references exactly one **Tool Bit**.
- A **Tool Bit** belongs to zero or more **Tool Libraries** (libraries are independent of **Jobs**).
- A **Dressup** wraps exactly one **Operation** (and is itself treated as an **Operation** in the **Job**'s output order).
- An **Operation** uses zero or more **Generators** to build its **Toolpath**.
- A **Job** + a **Post-Processor** + a **Machine** produce the final G-code output.
- **Sanity Check** runs over a **Job** and emits a report; it does not modify the **Job**.

## Example dialogue

> **New contributor:** "I want to add a chamfer pass after my profile. Do I add another **Operation**?"
> **Maintainer:** "You *can*, but if you want it tied to the same edges with the same depth/lead-in math, use a **Dressup** instead — wrap the **Profile** with a chamfer dressup so it stays in sync."
>
> **New contributor:** "And the **Tool Controller** — do I make a new one for the chamfer bit?"
> **Maintainer:** "Yes. The **Tool Controller** is the cutter *plus* its feeds and speeds. The **Tool Bit** is just the geometry — you can have two **TCs** pointing at the same chamfer **Tool Bit** if you want different feeds for roughing vs finishing."
>
> **New contributor:** "Where does the G54/G55 stuff get set?"
> **Maintainer:** "On the **Job**. We call those **Fixtures** — pick which ones to output and the post will emit a section per fixture."
